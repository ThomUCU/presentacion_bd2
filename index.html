BLOQUE 01 Motivación — ¿por qué existen los índices?

Sin índices, el DBMS realiza un Full Table Scan: recorre la tabla entera fila por fila para encontrar las filas que cumplen la condición. El costo crece linealmente con la cantidad de filas.

Con índices, el acceso puede pasar a ser sublineal — típicamente O(log n) con estructuras de tipo B-Tree — lo que representa una mejora radical en tablas grandes.

Principio fundamental: el objetivo de un índice es minimizar la cantidad de páginas de disco que el DBMS debe leer para resolver una consulta.
Silberschatz, Korth & Sudarshan — Database System Concepts, Cap. 11 "Indexing and Hashing". También: Ramakrishnan & Gehrke — Database Management Systems, Cap. 8.
BLOQUE 02 ¿Qué es un índice?

Un índice es una estructura de datos auxiliar que el DBMS mantiene sobre una o más columnas de una tabla, separada del dato en sí, para acelerar la localización de filas.

Clave del índiceEl o los atributos sobre los que se construye la estructura (ej: apellido, fecha_nac).
PunteroReferencia a la ubicación física (página de disco / RID) donde reside la fila real.
Estructura separadaEl índice NO es la tabla. Se mantiene sincronizado automáticamente por el DBMS ante INSERT / UPDATE / DELETE.
TransparenciaEl índice es transparente para el usuario/programador SQL. El optimizador decide si usarlo o no.
Analogía clásica: el índice al final de un libro. No releés el libro entero, vas directo a la página indicada.
BLOQUE 03 Tipos de índices
Tipo Descripción Cuándo usarlo
Clustered Los datos físicos están ordenados en disco según la clave del índice. Solo puede existir uno por tabla. PK, columnas usadas en ORDER BY frecuente, rangos (BETWEEN).
Non-clustered Estructura separada con punteros a los datos. Los datos no están ordenados físicamente según este índice. Pueden existir varios por tabla. FK, columnas en WHERE o JOIN frecuentes.
Único (UNIQUE) Garantiza que no existan valores duplicados en la columna indexada. Implica una constraint de unicidad. Claves candidatas, atributos con restricción de unicidad.
Compuesto Índice sobre dos o más columnas. El orden de las columnas importa: es útil para consultas que filtran por el prefijo izquierdo del índice. Consultas con múltiples condiciones en WHERE sobre las mismas columnas.
Bitmap Usa vectores de bits, uno por valor distinto. Muy eficiente para columnas de baja cardinalidad. Costoso en actualizaciones. Género, estado civil, país — columnas con pocos valores distintos. Típico en Data Warehouses.
Silberschatz — Cap. 11.1 "Basic Concepts". Ramakrishnan — Cap. 8.2 "B+ Tree Index". Coronel & Morris — Database Systems, Cap. 11.
BLOQUE 04 Índices y claves (PK / FK) — conexión con el curso

Este punto conecta directamente con el material del curso. En el Diseño Físico se consideran las "necesidades de estructuras de acceso, índices, dispersiones, bitmaps" como parte de la etapa posterior al Diseño Lógico.

✓ Clave primaria (PK)
  • El DBMS indexa la PK automáticamente (generalmente como índice clustered).
  • El material del curso indica: "los DBMS indexan la clave primaria".
  • Por eso se prefieren PKs simples: menor tamaño del índice, búsquedas más rápidas.
✓ Clave foránea (FK)
  • Las FK son candidatas naturales a índices non-clustered.
  • Se usan constantemente en JOIN entre tablas.
  • Un índice en la FK del lado "muchos" puede cambiar drásticamente el costo del JOIN.
Material del curso — BDII_ML_01 Diseño Lógico: "Eficiencia en los accesos, por lo general los DBMS indexan la clave primaria" y "Necesidades de estructuras de acceso, índices, dispersiones, bitmaps".
BLOQUE 05 Cuándo usar índices — y cuándo no
✓ Conviene indexar
  • Columnas en WHERE con alta selectividad (pocos resultados).
  • Columnas usadas en JOIN frecuentes (especialmente FK).
  • Columnas en ORDER BY o GROUP BY.
  • Tablas grandes con lecturas frecuentes.
✗ No conviene / perjudica
  • Columnas de baja cardinalidad (booleanos, pocos valores → usar Bitmap si aplica).
  • Consultas que devuelven gran porcentaje de la tabla (full scan puede ser más barato).
  • Tablas pequeñas.
  • Tablas con muchas escrituras (INSERT/UPDATE/DELETE frecuentes).
Costo de mantenimiento: cada modificación de datos obliga al DBMS a actualizar también todos los índices de esa tabla. Índices innecesarios penalizan las escrituras.
BLOQUE 06 Sintaxis SQL 
-- Índice simple sobre una columna
CREATE INDEX idx_apellido ON Empleado(apellido);

-- Índice con restricción de unicidad
CREATE UNIQUE INDEX idx_ci ON Empleado(ci);

-- Índice compuesto (columna1 es el prefijo izquierdo)
CREATE INDEX idx_dep_sal ON Empleado(departamento, salario);

-- Eliminar un índice
DROP INDEX idx_apellido;

-- Ver plan de ejecución (PostgreSQL / MySQL)
EXPLAIN SELECT * FROM Empleado WHERE apellido = 'García';

En SQL Server se usa CREATE CLUSTERED INDEX / CREATE NONCLUSTERED INDEX. En Oracle se usa CREATE INDEX ... BITMAP para índices bitmap.

BLOQUE 07 Plan de ejecución y Query Optimizer

El Query Optimizer es el componente del DBMS que analiza la consulta SQL y decide cómo ejecutarla: si usa un índice, qué tipo de join aplica, en qué orden accede a las tablas. El programador escribe qué quiere; el optimizador decide cómo obtenerlo.

EXPLAIN / EXPLAIN PLAN: comando que muestra el plan de ejecución elegido por el optimizador sin ejecutar realmente la consulta. Permite diagnosticar si se están usando índices o no.

Términos importantes que pueden aparecer en un plan de ejecución:

Seq Scan — full table scan Index Scan — usa el índice Index Only Scan — solo lee el índice Nested Loop — join anidado Hash Join — join por hash Merge Join — join por orden
Si un índice existe pero el optimizador no lo usa, puede ser porque la tabla es pequeña o la selectividad es baja. El optimizador calcula el costo estimado de cada alternativa y elige la menor.
Silberschatz — Cap. 12 "Query Processing". Ramakrishnan — Cap. 12 "Evaluation of Relational Operators".
BLOQUE 08 Relación con el diseño conceptual y lógico

Los índices pertenecen al Diseño Físico, que es la etapa posterior al Diseño Lógico. Es la única etapa que es dependiente del motor de base de datos seleccionado.

Diseño Conceptual (MER)Define entidades, relaciones, atributos e identificadores. Un buen modelado facilita identificar qué columnas serán críticas.
Diseño LógicoSe baja el MER a tablas. Las PK y FK quedan definidas. El DBMS indexará automáticamente la PK.
NormalizaciónReduce redundancia pero puede generar más JOINs. Los índices en FK compensan ese costo adicional de lectura.
Diseño FísicoAquí se crean índices adicionales según los patrones de acceso reales. Se consideran distribución en disco, dispersiones, bitmaps y demás parámetros del DBMS.
